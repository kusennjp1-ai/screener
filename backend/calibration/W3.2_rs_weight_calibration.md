# W3.2 — RS recency-weight calibration (real-data run)

Data-driven answer to the audit question flagged on `RelativeStrengthCalculator.PERIODS`:
does the 40/20/20/20 recency tilt actually predict forward outperformance, or is it
intuition at overfit risk? This run answers it on **real Yahoo OHLCV**, not fixtures or
the synthetic stand-in.

## How this was produced

```bash
cd backend
# 1) fetch real 5y daily OHLCV for the US ex-ETF calibration universe + SPY
PYTHONPATH=. python3 scripts/fetch_us_universe.py \
    --symbols-file scripts/us_universe.txt --out-dir realdata
# 2) walk-forward calibrate the candidate recency-weight configs vs SPY
PYTHONPATH=. python3 scripts/calibrate_rs_weights.py \
    --source csv --data-dir realdata --benchmark spy --top-frac 0.30
```

- **Source:** real Yahoo Finance daily OHLCV (5y), one CSV per symbol. The raw
  `realdata/` CSVs are `.gitignore`d (bulk market data); this report is the committed record.
- **Universe:** 136 usable US ex-ETF names + SPY benchmark (2 of 139 requested
  symbols — `MMC`, `FI` — failed to fetch: delisted / ticker change at the vendor).
- **Method:** 8 walk-forward as-of dates; at each, rank every name by each config's
  recency-weighted relative outperformance vs SPY, flag the top 30% as "RS leaders",
  and record their forward excess return (stock − SPY) at T+1 / T+5 / T+21.

## Decision

**Recommended: `baseline_40_20_20_20` — KEEP the current default.**

No challenger beat the baseline on Sharpe **and** win-rate across a majority of horizons
with sufficient n (the conservative decision rule in `calibrate_rs_weights.decide`). The
recency-heavier configs edge the baseline slightly at T+1 (Sharpe +0.40/+0.41 vs +0.36)
but none of the differences clear the margin thresholds, and the ranking is not consistent
across horizons — exactly the pattern the conservative rule is designed to reject as
overfitting.

### Effect on code

`RelativeStrengthCalculator.PERIODS` stays at the calibrated winner:

```python
PERIODS = {63: 0.40, 126: 0.20, 189: 0.20, 252: 0.20}  # 40/20/20/20
```

The recommended config **is** the current default, so the `periods` default needs **no
change**. The constructor remains `periods`-overridable, so a future re-run on a larger
universe / more as-of dates can inject a validated winner with no code edit.
